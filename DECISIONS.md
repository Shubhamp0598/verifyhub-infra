## Phase 0 — Repo structure

**Decision:** Split into `infra/modules` (reusable, environment-agnostic) and
`infra/environments/{dev,staging,prod}` (thin root modules supplying config only).

**Why:** Standard pattern for "one codebase, many environments." Modules hold logic
(how to build a VPC). Environment folders hold data (how big, how many). A new
environment is a new folder + tfvars file — no copy-pasted resource blocks to drift.

**Assumption:** The `/app` stub referenced in the assignment was not included in the
materials I received (only the PDF brief). I'm proceeding by defining a generic
container contract (health check path, port, env vars) that the ECS module is built
against, documented in `app/README.md`, so swapping in a real stub later is a config
change, not a redesign.

**Assumption:** Single AWS account for dev/staging/prod, isolated via separate state
files, VPCs, and IAM boundaries rather than separate AWS accounts. Revisited under
Security — for regulated KYC/PII data, account-per-environment via AWS Organizations
would be the production-grade choice; noted as a "what I'd do next."

**Region:** us-east-1.

## Phase 1 — Networking + state

**S3 backend locking:** Using Terraform 1.10+ native S3 lockfile locking
(`use_lockfile = true`) instead of the older DynamoDB-based locking. Removes a
stateful dependency (no lock table to manage/pay for) since S3 now supports
conditional writes natively.

**Subnet layout:** 3-tier subnets per AZ — public (ALB/NAT), private-app (ECS),
private-data (RDS/cache, no internet route). Per-AZ NAT gateways instead of one
shared NAT — costs more but an AZ outage doesn't take out the other AZ's egress.
Could go single-NAT for dev to save cost; kept symmetric for now.

## Phase 3 — Data layer

- DB password generated by Terraform (`random_password`), stored only in Secrets
  Manager — never in tfvars or CLI args. App reads it at runtime via the execution
  role's `secretsmanager:GetSecretValue` permission (Phase 2).
- Document bucket has no public access at any layer; upload/download goes through
  presigned URLs from the Document Service, not bucket ACLs. Lifecycle rule expires
  objects after ~7yr in prod / 90d in dev — placeholder retention, real number should
  come from actual KYC compliance requirements, not an engineering guess.
- SQS has a DLQ with `maxReceiveCount=5` — failed verification jobs land here instead
  of retry-looping forever. DLQ depth becomes an alarm target in Phase 6.
- Redis: single node in dev/staging, 2-node with automatic failover in prod.
  Encrypted at rest and in transit everywhere.

## Phase 4 — Circular dependency: ECS service SGs vs DB/cache SGs

**Issue found while wiring dev:** the original `ecs-service` module created its own
security group internally. RDS/ElastiCache modules need to allow ingress *from*
each service's SG. But each service also needs the DB endpoint as an env var to even
start. That's a cycle: `ecs-service` → database (for endpoint) → security group →
`ecs-service` (SG owned inside the module). Terraform can't resolve a cycle like this
within one apply.

**Fix:** moved security group ownership out of the `ecs-service` module and into the
environment root as plain `aws_security_group` resources with no dependency on the
ECS service itself. `ecs-service` now takes `security_group_id` as an input instead
of creating one. Database/cache modules reference these SG ids directly for ingress
rules — since an SG resource has no dependency on the service running inside it, the
cycle breaks cleanly.

**Why this matters:** this is the kind of dependency graph mistake that's easy to
introduce when building modules bottom-up (a module looks fine in isolation) and
only shows up once everything is wired together at the environment level — which is
why this repo deliberately built and fmt/validate-checked lower-level modules first,
then did the full environment wiring as its own explicit step, rather than writing
everything in one pass.

## Phase 4 — Blast radius: single account now, multi-account as future work

**Decision:** dev/staging/prod share one AWS account, isolated by separate VPCs,
separate state files, and separate IAM roles per service — not separate AWS
accounts.

**The question behind "why not multi-account":** account-per-environment exists to
answer one thing — if something goes wrong (a compromised credential, a bad IAM
policy, a runaway script, a misconfigured resource), how far does the damage spread
before it hits a hard wall? A VPC boundary stops network traffic. An IAM policy
stops an API call. Neither stops a stolen root credential, a misissued IAM policy
with an unintended wildcard, or a service-quota/billing blast radius from one
environment starving another. Only an AWS account boundary stops those, because
account boundaries are enforced by AWS itself, not by anything we wrote in Terraform
that could itself contain a bug.

Concretely, in the current single-account design: a bug in the `worker` task role
policy that's too permissive is contained by IAM, correctly, in this design. But an
IAM boundary misconfiguration that grants dev worker access to a prod resource ARN
by typo, or a leaked dev AWS credential with broader account-level permissions than
intended, is not contained — it's a same-account blast radius. That's the specific
gap multi-account closes.

**Why single-account for this exercise:** the assignment scope is demonstrating
infra design end-to-end (networking, compute, data, security, CI/CD, observability)
within a bounded timeframe, on infrastructure with no live traffic or real user
data. Multi-account is mostly an operational/organizational decision (AWS
Organizations setup, cross-account IAM roles for CI/CD to assume, shared VPC or
Transit Gateway if cross-env traffic is ever needed, centralized logging via a
dedicated log-archive account) — real work, but it doesn't change how any individual
service's Terraform is written. It's additive scaffolding around this repo, not a
rewrite of it.

**What changes if we go multi-account later:** each environment's `backend.hcl`
points to a state bucket in that environment's own account. The GitHub Actions
workflow assumes a per-environment IAM role via OIDC instead of one shared set of
credentials — so a compromised CI token for staging cannot touch prod. AWS
Organizations SCPs (Service Control Policies) enforce guardrails per account (e.g.
"prod account cannot disable CloudTrail," "dev account cannot create resources
outside us-east-1") as an outer layer that doesn't depend on any engineer
remembering to apply it via Terraform. This is what I'd build next if this were a
real production KYC platform handling regulated PII, given more time.

## Phase 5 — CI/CD

**Pipeline shape:** Plan-on-PR (posted as a PR comment) + apply-on-merge, sequential
dev → staging → prod via job `needs`. Prod is gated by a required GitHub Environment
reviewer — the last human checkpoint even though the pipeline is otherwise fully
automated.

**Credentials (known gap):** using long-lived AWS credentials as GitHub Secrets for
now, not OIDC federation. OIDC removes the need to store any AWS credential in
GitHub at all (GitHub issues a short-lived token, AWS trusts GitHub's OIDC provider
directly) — this is what I'd switch to next; kept as secrets here for setup speed
given the deadline.

**IAM permissions (known gap):** the GitHub Actions IAM user was granted `PowerUserAccess` and, separately, `IAMFullAccess` (PowerUserAccess deliberately excludes IAM actions as a guardrail against privilege escalation — adding IAMFullAccess on top knowingly bypasses that guardrail) to unblock apply runs during development. This is broad and would
not be acceptable for a real deployment pipeline — the correct design is a
purpose-built deploy role scoped to exactly the services this Terraform manages
(ECS, RDS, S3, SQS, ElastiCache, IAM within a defined path, VPC/EC2 networking),
assumed via GitHub OIDC rather than long-lived credentials. Documented here rather
than fixed immediately, given time constraints — **this is the first thing I'd lock
down before treating this pipeline as production-ready.**

**Verified end-to-end, not just reviewed as YAML:** opened a real PR to confirm the
pipeline actually works. Caught two real issues this way: unformatted tfvars files
(`terraform fmt -check` correctly failed the build) and deprecated action runtime
versions (bumped `checkout`/`configure-aws-credentials`/`github-script` to current
majors). Confirmed plan-on-PR with posted comments works across all three
environments.

## Phase 6 — SLOs

Three SLOs chosen to map directly to what a customer business actually feels:

1. **API availability: 99.5% of requests to api-gateway return non-5xx over a
   rolling 30-day window.** Chosen over a stricter 99.9% because this is a KYC
   integration point, not a payment rail — a brief outage delays a verification, it
   doesn't lose money mid-transaction. 99.5% allows ~3.6 hours/month of error budget,
   realistic for a single-region deployment without multi-region failover (out of
   scope here).

2. **Verification job completion: 95% of jobs complete (pass/fail result written)
   within 2 minutes of upload, end to end.** This is the actual product promise to
   client businesses — "how long until I get an answer" — not an infra-internal
   metric. Derived from the worker's queue-depth-based autoscaling target: at
   backlog-per-task=10 with typical processing time, 2 minutes is achievable without
   over-provisioning workers at rest.

3. **API p95 latency: under 500ms for api-gateway responses** (excludes the async
   verification result, which is covered by SLO 2). Covers the synchronous "did my
   request get accepted" path, separate from async processing time.

**What I'd do differently with more time:** these numbers are reasoned estimates,
not derived from real traffic data (none exists yet — this is a from-scratch
build). In a real rollout I'd set looser initial SLOs, measure actual behavior for
2–4 weeks, then tighten based on real p50/p95/p99 data rather than guessing upfront.

## Real-apply findings

dev and staging were applied live against a personal free-tier AWS account — beyond
what the brief required ("no live deploy is required") — specifically to stress-test
the code past static `validate`/`plan`. This caught three real bugs that static
checks alone would have missed:

### 1. Target group had no listener

**Error:** `target group ... does not have an associated load balancer`.

**Cause:** target groups (one per public-facing service) and the ALB were both
created, but nothing attached them together — ECS refuses to register a service
against a target group with no listener. This was a known gap flagged when the
`ecs-cluster` module was first written, but only the HTTPS/cert piece was
consciously deferred — the base HTTP listener was mistakenly never added at all,
and it only surfaced on a real apply, not during validate/plan.

**Fix:** added `infra/environments/dev/listeners.tf` — one HTTP listener on `:80`,
default action forwards to `api-gateway` (primary public entry point per the
brief), with path-based rules routing `/admin/*`, `/dashboard/*`, `/documents/*` to
the other 3 services. All 4 public-facing services share one ALB rather than one
each, to keep cost/complexity down for what's currently low-traffic internal-facing
surfaces.

**Still deferred, real gap:** no HTTPS/ACM cert — no real domain exists for this
exercise. Production needs a Route53 zone, an ACM cert, a `:443` listener, and an
HTTP→HTTPS redirect rule on `:80`.

### 2. ECS service creation raced the ALB listener

**Error:** same "target group has no associated load balancer" error dev hit
earlier — but dev's apply had *already succeeded* with the same `listeners.tf` code.

**Root cause:** nothing forced ECS service creation to wait for the ALB listener.
Listener rules correctly depend on target groups (referenced by ARN), but nothing
in the reverse direction told the ECS service to wait for the listener/rule to
exist. Terraform parallelizes independent resources by default — dev's apply
happened to have the listener finish first by API timing luck; staging's didn't.
This was a latent bug in dev's "successful" apply too, just not yet triggered.

**Fix:** added explicit `depends_on = [aws_lb_listener.http]` (or the relevant
listener rule) to each of the 4 public-facing `ecs-service` module calls, in all
three environments. Removes reliance on timing luck.

### 3. Free-tier RDS backup retention ceiling is 1 day, not 7

**Error:** `FreeTierRestrictionError` on staging (default was 3) and would have hit
the same on prod (default was 7) — same AWS account across all three environments,
so the free-tier ceiling applies uniformly regardless of environment.

**Fix:** set `db_backup_retention_days` default to 1 in dev, staging, and prod for
now. This is a real gap versus what a production system should have (prod should
run un-throttled retention, likely 7–35 days depending on compliance needs) — it's
an artifact of building/testing this on a personal free-tier AWS account rather
than a proper production-tier account. Documented here rather than hidden; the
Terraform variable already supports a higher value — this is purely an
account-tier constraint, not a design limitation.

### Note — live apply of prod was not completed, by design

dev and staging were applied successfully as described above. prod's apply failed
on AWS account-level quota limits (Elastic IP limit, RDS instance limit) tied to
using a single personal free-tier AWS account for all three environments — not a
code defect. `terraform validate` and `terraform plan` both pass cleanly for prod
with 0 errors, which is what the assignment requires; live apply was never a
requirement ("No live deploy is required" per the brief) and was only attempted
opportunistically for dev/staging as extra verification.

## Honest gaps and caveats, re-reading the brief

**Region portability:** the codebase claims "adding a new region should be cheap"
via `var.aws_region` and per-environment `backend.hcl`/tfvars. This is true in
principle (no hardcoded region in any module), but was never actually tested
end-to-end — availability zones (`us-east-1a`/`1b`) are hardcoded per-environment
tfvars rather than data-sourced, so a new region would need new AZ names supplied,
not zero-touch. A real region rollout would use a `data "aws_availability_zones"`
lookup instead of hardcoded AZ strings to make this fully automatic.

**CI/CD process fidelity:** the intended flow is PR → plan comment → review →
merge → apply. During active debugging, several fixes were pushed directly to
`main` to unblock forward progress faster, bypassing the PR review step that the
rest of this repo argues for. This is a real inconsistency between the stated
process and what actually happened during development — noted here rather than
edited out of the commit history, since the assignment explicitly said commit
history and reasoning matter more than a polished-looking end state.

**Live apply, in hindsight:** dev and staging were applied for real against personal
AWS free-tier credentials, beyond what the assignment required. This surfaced and
fixed real bugs (an ALB listener race condition, a missing `depends_on`, a free-tier
RDS retention limit) that static plan/validate alone would not have caught — a
genuine benefit — but wasn't necessary for the deliverable, and in hindsight the
time would have been better spent on diagrams and docs polish earlier. All live
resources have since been destroyed; only the code remains.

## What I'd Do Next — Consolidated

Everything below was deliberately deferred, not overlooked. Each item maps to a
specific rubric area in the brief, with the tradeoff that justified deferring it
and what the actual next step would be.

### Container supply chain

**Current state:** ECR with scan-on-push and immutable tags — catches known CVEs
in an image and stops a tag from being silently overwritten after review.

**Gap:** scanning on push only tells you a problem exists; nothing today *blocks*
a deploy because of it, and nothing verifies the image running in ECS is the exact
image that was scanned and reviewed.

**What I'd add, in order of priority:**
1. **Gate the CI/CD pipeline on scan results** — fail the `apply` job (not just
   `plan`) if ECR reports a CRITICAL/HIGH vulnerability, rather than scanning being
   informational only.
2. **Image signing (Sigstore/cosign)** — sign images at build time, verify the
   signature at deploy time. Answers "is this the exact artifact that was built and
   scanned by CI, not something swapped in between"? — this is the actual
   supply-chain guarantee; scanning alone doesn't provide it.
3. **Base image pinning + automated update PRs** (Renovate/Dependabot) — otherwise
   a "clean" scan today can silently rot as new CVEs are disclosed against a base
   image that never gets rebuilt.

**Tradeoff for deferring:** each of these is orthogonal to the Terraform itself
(they live in Dockerfiles and CI config, not `infra/`), so they don't block the
core deliverable, but they're real gaps in "container supply chain," one of the
brief's explicitly scored security areas.

### Observability: beyond the 3 SLOs

**Current state:** 3 SLOs, CloudWatch alarms per SLO, DLQ depth, RDS health,
per-service CPU, one SNS topic, one dashboard.

**Gap:** alarms tell the on-call engineer *that* something is wrong; the brief
asks whether they can tell **what and why at 3am** — that needs correlated signals,
not just threshold breaches.

**What I'd add:**
1. **Structured JSON logging** across all 4 services, shipped to CloudWatch Logs
   Insights (or an ELK/OpenSearch stack) with a consistent schema (`request_id`,
   `service`, `verification_id`) so a single verification can be traced across
   API Gateway → SQS → Worker → Document Service by one correlation ID.
2. **Distributed tracing (AWS X-Ray or OpenTelemetry)** — the request flow crosses
   4 services and a queue; without tracing, "why is job completion SLO breached"
   requires manually correlating logs across services by timestamp, which is slow
   and error-prone exactly when speed matters most (a live incident).
3. **Runbooks** — one per alarm, answering: what does this alarm firing actually
   mean, what's the likely cause, what's the first command to run, who/what to
   escalate to. An alarm without a runbook still leaves the on-call engineer
   guessing, which is precisely the 3am scenario the brief calls out.
4. **Log retention & cost control** — no retention policy is set today; CloudWatch
   Logs default retention is "never expire," which is both a cost risk and, for
   KYC data potentially appearing in logs, a compliance risk (PII in logs with no
   expiry).

**Tradeoff for deferring:** tracing and log pipelines add real infra (OpenSearch
domain or X-Ray, additional IAM, additional cost) and were judged lower priority
than getting core compute/data/security/CI-CD working end-to-end first, given the
scope of a single assessment. This is the single biggest gap versus a real
production on-call posture.

### Secrets lifecycle

**Current state:** all secrets in Secrets Manager, KMS-encrypted, least-privilege
IAM access to read them, never in tfvars or CLI args.

**Gap:** storage is solved; rotation and compromise-response are not.

**What I'd add:**
1. **Automatic rotation** for the DB password via Secrets Manager's native RDS
   rotation Lambda — currently the password is generated once by
   `random_password` and never rotates.
2. **A documented break-glass procedure** — if a secret is suspected leaked
   (e.g., committed to a fork, exposed in a log line), what's the actual sequence
   to rotate it and invalidate the old value across every service that holds a
   cached reference?

**Tradeoff for deferring:** rotation adds a Lambda + additional IAM surface per
secret type — real complexity for a system with no live traffic today. Worth
doing before this ever holds real KYC/PII data.

### Networking: future work

**Current state:** 3-tier subnets per AZ, per-AZ NAT gateways (symmetric across
all three environments), no internet route from the data tier.

**Gap 1 — dev cost:** per-AZ NAT for dev is arguably over-provisioned; dev doesn't
need AZ-outage resilience for its own sake. **Tradeoff:** kept symmetric with
staging/prod so dev stays a faithful, low-surprise staging ground for infra
changes before they hit prod — asymmetric environments make "does this work in
prod" harder to answer from dev testing alone. Would revisit if NAT cost became
material.

**Gap 2 — cross-environment/cross-account networking:** not needed today since
all three environments live in one account with fully isolated VPCs. If we move
to multi-account (see Phase 4), and any service ever needs to reach across
environments (e.g., a shared internal tool), that requires either VPC peering or
a Transit Gateway. Deferred because nothing today requires cross-environment
traffic — this is speculative future-proofing, not a current gap.

### Extensibility to new *services* (not just environments)

The brief specifically asks: "a change stays localized, reviewable, and low risk,
and a teammate could extend it without fear" for **new services**, not just new
environments — this is a distinct claim from "adding an environment is cheap" and
deserves its own answer.

**Current state:** adding a 5th service today means: add an ECR module call, add
an `ecs-service` module call with its own task role/SG, add a target group +
listener rule if it's public-facing, wire any new queue/cache/DB access explicitly
per-service (least privilege, not shared credentials). Each of these is additive —
no existing service's Terraform changes — because of the module boundaries set in
Phase 0 and the SG-ownership fix from Phase 4.

**What's not yet proven:** this claim has never actually been exercised — no 5th
service has been added end-to-end as a test. That's a real gap: "structured so it
*should* work" is weaker evidence than "we added one and it took N lines in M
files." **Next step:** actually add a placeholder 5th service (e.g., a
notifications service) as a trial run, and record how many files changed and
whether any existing service's plan showed unexpected diffs — that's the real
proof of "low risk to extend."

### Cost

**Not addressed anywhere else in this doc — a real gap**, since cost is an
implicit tradeoff behind several decisions already made (per-AZ NAT, per-AZ
ElastiCache failover in prod, retention periods) without ever being named as such.

**What I'd add:**
1. **Cost estimation in CI** — run `infracost` (or similar) against each
   environment's plan and post estimated monthly cost as a PR comment alongside
   the Terraform plan, so a reviewer sees "this change adds $X/month" at review
   time, not after the bill arrives.
2. **Environment-level cost tradeoffs, made explicit:**
   - **NAT gateways:** per-AZ in all three environments today (symmetric, see
     Networking above) — dev/staging could go single-NAT to cut ~2/3 of NAT cost,
     traded against dev/staging no longer mirroring prod's resilience profile.
   - **ElastiCache:** single-node in dev/staging vs. 2-node w/ failover in prod —
     already cost-optimized; dev/staging accept a cache outage as acceptable risk
     for non-production traffic.
   - **RDS:** no reserved instances or Savings Plans considered yet — for a
     workload with predictable prod baseline load, 1- or 3-year RDS reserved
     instances would meaningfully cut the largest single line item, traded against
     reduced flexibility to resize/change instance class.
   - **Fargate vs. EC2-backed ECS:** Fargate was chosen (implicitly, via the
     `ecs-service` module) for operational simplicity — no EC2 fleet to patch or
     right-size — traded against Fargate's per-vCPU/GB premium over EC2 at
     steady-state, sustained load. Worth revisiting for the Verification Worker
     specifically if its bursty load pattern, once measured for real, shows long
     sustained-high periods rather than true bursts — EC2 Spot for the worker
     tier could be materially cheaper for a queue-driven, restart-tolerant
     workload.
3. **Tagging strategy for cost allocation** — no consistent `environment`/
   `service`/`cost-center` tags enforced today across modules; without this, a
   real AWS Cost Explorer breakdown by service/environment isn't possible even
   though the infrastructure is already logically separated that way.

**Tradeoff for deferring:** cost tooling and reserved-capacity decisions are most
useful once there's real, measured traffic to size against — doing this
prematurely on invented numbers would be guessing twice (once on the SLOs, again
on capacity). Tagging, however, is cheap to add now and should probably have been
there from Phase 0 — that one's a genuine oversight, not a considered tradeoff.
